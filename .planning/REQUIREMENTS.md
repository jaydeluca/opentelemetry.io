# Requirements: OpenTelemetry Configuration Documentation Sync

**Defined:** 2026-03-06
**Core Value:** Documentation for OpenTelemetry SDK configuration must always match the source schema definitions

## v1 Requirements

Requirements for initial release. Each maps to roadmap phases.

### Core Functionality

- [x] **CONF-01**: Python script reads YAML schemas from opentelemetry-configuration repo
- [x] **CONF-02**: Python script parses YAML into structured data (types, properties, enums, constraints)
- [x] **CONF-03**: Python script generates language implementation status tables (accordion format with per-language tables)
- [x] **CONF-04**: Python script generates type documentation tables (type descriptions, properties, constraints)
- [x] **CONF-05**: Python script updates markdown files between BEGIN/END markers

### GitHub Automation

- [ ] **CONF-06**: GitHub Action runs on manual dispatch
- [ ] **CONF-07**: GitHub Action runs on schedule (daily at 03:00 UTC)
- [ ] **CONF-08**: GitHub Action runs when opentelemetry-configuration releases
- [ ] **CONF-09**: GitHub Action creates/updates PR with changes using otelbot
- [ ] **CONF-10**: Fork testing workflow allows contributors to test in their forks

### Quality & Integration

- [x] **CONF-11**: Script follows collector-sync structure (uv project, src/tests, npm integration)
- [x] **CONF-12**: Generated content formatted correctly (runs `npm run fix:format` after generation)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Real-time sync on every schema commit | Creates noise with incomplete changes, scheduled/release triggers are sufficient |
| Custom templating languages (Jinja2, Mako) | Adds complexity, f-strings are clearer and sufficient for markdown generation |
| Multi-language page generation | English-only pages use Hugo fallback for localization, automation doesn't need to handle it |
| Interactive schema browsers | Hugo static site can't support dynamic browsing, markdown documentation is sufficient |
| Automatic deprecation detection | Schema doesn't contain deprecation metadata, manual marking required |
| Inline schema examples | Examples live in opentelemetry-configuration/snippets, not generated by automation |
| Schema evolution tracking | Version control (git) handles this, automation doesn't need to duplicate it |
